# Commit Skill

변경사항을 분석해 Conventional Commit 메시지를 작성하고 자동으로 커밋합니다.

## 커밋 절차

1. 현재 상태를 한 번에 파악합니다. 아래 명령을 함께 실행해 스테이징 여부, 수정 파일, untracked 파일, 최근 커밋 스타일을 확인하세요.
   - `git status`
   - `git diff --staged` (스테이징된 변경)
   - `git diff` (스테이징 안 된 변경)
   - `git log --oneline -10` (이 저장소의 커밋 스타일 참고)
2. 스테이징된 변경이 하나도 없으면, 이번 작업과 관련된 파일을 `git add` 로 스테이징합니다. 관련 없는 파일(설정 파일, 빌드 산출물 등)을 무분별하게 함께 담지 마세요.
3. 스테이징된 내용을 분석해 아래 규칙대로 메시지를 작성하고 `git commit` 으로 바로 커밋합니다. 메시지 작성 후 사용자 확인을 따로 받지 않습니다.
4. 커밋 후 `git log --oneline -1` 로 결과를 보여줍니다.

## 커밋 메시지 규칙

형식: `<type>(<scope>): <설명>`

- **type** (영어, 소문자):
  - `feat` — 새로운 기능
  - `fix` — 버그 수정
  - `refactor` — 동작 변화 없는 코드 구조 개선
  - `test` — 테스트 추가·수정
  - `docs` — 문서 변경
  - `chore` — 빌드·설정·도구 등 그 외 잡일
- **scope** (선택, 영어): 변경된 주요 모듈·컴포넌트 이름. 변경 범위가 넓어 특정하기 어려우면 생략 가능합니다.
- **설명** (한국어): 무엇을 했는지 현재형으로 간결하게. 마침표를 붙이지 않습니다. "~함/~추가" 같은 종결형이 자연스럽습니다.

이 저장소의 실제 커밋 기록(`feat: 카테고리별 Todo 필터 추가`, `docs: README에 개발 서버 실행 명령어 추가`)과 같은 스타일을 따르세요. type은 영어, 설명은 한국어입니다.

## 판단 가이드

- 한 커밋에는 하나의 논리적 변경만 담는 것이 이상적입니다. 서로 다른 성격의 변경이 섞여 있으면 사용자에게 분리 커밋을 제안하세요.
- type이 애매할 때: 사용자에게 새 가치를 주면 `feat`, 기존 동작을 고치면 `fix`, 둘 다 아니고 동작이 그대로면 `refactor` 입니다.
- 본문(body)이 필요할 만큼 변경이 복잡하면 제목 아래 빈 줄 뒤에 "왜" 변경했는지 한국어로 덧붙이세요. 대부분의 작은 변경은 제목 한 줄로 충분합니다.

## 예시

- `feat(todo): 완료 상태별 필터 탭 추가`
- `fix(todo-item): 체크박스 토글이 저장되지 않던 문제 수정`
- `refactor(api): 할 일 조회 로직을 서버 컴포넌트로 이동`
- `docs: README에 개발 서버 실행 명령어 추가`
- `chore: shadcn UI 수정 규칙 추가`
